Case Study

Data Explorer

UX / Data Visualization

Fluxx Grantmaker is a cloud-based relational database software for nonprofit organizations, which supports deep customization and serves dozens of different user roles. Think Salesforce levels of complexity. There are custom record types, bespoke workflows, and a primary UI that’s akin to a super-charged Kanban board, with hundreds of interrelated records visible at once.

The primary unit of Grantmaker’s UI is a “card”, a filtered list of records. Most daily users build up dashboards of cards, laid out in a specific order to support their approval workflows. These cards can be simple list views, tables, summary statistics, or data visualizations.

Advanced users have access to robust reporting tools, but when most staff need answers to questions like “How much total giving has your program done so far this year?”, many were lost and frustrated. In interviewing dozens of daily non-admin users, we repeatedly heard complaints like:

“My admin only allows us to run reports, not create them. Often the data I want is missing, but I don’t know how to get to it”
“I have some access to reporting, but it’s intimidating. There are hundreds of fields on each record. I don’t know where to start to find my data.”
“Sometimes I’ll export everything and just delete 90% of it, it’s easier than trying to build out what I need”

As an in-house product designer, I set out to re-factor Grantmaker's existing card views to support these kinds of requests.

Bridging the Gap

Two of Grantmaker’s existing card views provided an ideal starting point, though they were both in need of a major overhaul. The table view showed the data, sans styling and other distracting clutter. A perfect place to begin an export. But the columns are fixed, determined by administrators. The chart view gives trend insights at a glance, but the charts themselves are more flash than substance, and suffering from a serious lack of flexibility.

Helping our daily users find and deliver simple metrics about their records was a matter of combining multiple interfaces that were formally different but categorically the same. Building a chart summary doesn’t instill much confidence if you can’t see the data underneath. And rendering exported data just to delete it later is a waste of time, both on the server side and for the user.

Solution

The revamped data visualizations addressed a number of design problems that kept the old “chart view” from being truly useful. Here’s how:

  • Making charts easier to read with direct labeling
  • Reducing the superfluous use of color
  • Dynamic timespans for quick data exploration
  • Re-flow of controls
  • Summary statistics for simple double-checking (does this add up to what I’d expect? Are my filter settings right?)
