Case Study

SaaS Design System

Fortifying and standardizing UX patterns across a complex productivity platform

Product
Fluxx Grantmaker
Role
Senior Product Designer
Impact
  • Achieved WCAG AA 2.2 compliance
  • 20% Increase in development velocity
  • 30+% Increase in code coverage

Overview

Fluxx is the world's most popular SaaS platform for grants management. As an in-house senior product designer, I led the creation of a design system to standardize UI and UX patterns for a SaaS productivity product with more than 500 enterprise clients and tens of thousands of MAUs.

Problems

  • Highly customizable UI is difficult to document and audit
  • Lack of consistent information hierarchy makes content hard to scan
  • Front-end styles are fragmented and difficult to maintain

Goals

  • Modernization of UI look & feel
  • Accessibility improvements (visual & semantic)
  • Standardization of common UI elements
  • Streamlined development pipeline for new features

Process

In preparation for the development of a design system, my design partner and I started with a 2-week design sprint. I audited the existing app for usability issues, explored product parallels, and mocked up "ideal" states for the most common UX patterns.

The sheer complexity of Grantmaker as a platform led to the creation of "contextual variants" for certain design components in the system.

Using a text input as an example: public-facing forms have extremely high requirements for legibility, scan-ability, click target size, and multiple semantic encodings. On a table-like interface for site administrators, a low-friction approach with implied labelling makes much more sense. The design system maintains a common vocabulary between these disparate use cases while respecting the requirements of each.

The results of the design sprint were socialized with product and technical leadership, building momentum for to the creation of the platform's first full-fledged design system. Following Atomic Design principles, I developed:

  • Spacing, colors, and typography specifications (maintained in Figma)
  • Framework-agnostic design tokens (as JSON managed via Git)
  • React components (developed and QA'd via Storybook)

Instead of pushing forward the design system as a monolithic project, the design team collaborated with product managers to isolate upcoming new features to start building re-aligned components opportunistically. This way, every sprint included work to flesh out the React library of design system components and patterns.

I worked closely with engineers to fine-tune the structure of the built prototypes, and contributed usability tweaks to the developed prototypes directly via Git. I also designed a documentation site for Fluxx engineers, with live examples of React components and exportable code snippets.

Solution

After designing and compiling hundreds of individual patterns and variants, the Fluxx design system is now in regular use, powering dozens of shipped features. In six months, the system:

  • Achieved WCAG 2.2 AA compliance for the platform
  • Improved code coverage for new features by more than 30%
  • Reduced average story points for new features by 20%

Further Considerations

While components built with the new design system have been well-received by users, I would have liked more time in the process to put more holistic examples of design system patterns through user acceptance testing. The incremental approach that Fluxx favored was a business necessity, but being able to build a brand-new front end would have had a much bigger impact on the user experience, and would have allowed for true A/B testing of new and old components.
